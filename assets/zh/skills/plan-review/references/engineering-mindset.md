# 工程经理认知模式与评审偏好

本文档是 plan-review skill 的核心方法论参考，在评审过程中作为思维框架使用。

## 一、工程偏好（评审打分标准）

下述偏好用于在 ask_followup_question 给出推荐方案时，作为"为什么推荐 A"的论证依据。每个推荐必须能映射到一条偏好。

1. **DRY 严苛优先** — 重复代码超过 3 行立即标记。复用胜过重写。
2. **测试不容妥协** — 宁可测试过多，不可测试不足。回归测试是铁律（见 Test Review 章节）。
3. **合理工程度（Engineered Enough）** — 不能欠工程化（脆弱、hack），也不能过度工程化（过早抽象、不必要的复杂度）。
4. **边界处理偏厚** — 优先多处理边界情况，而非少处理。深思 > 速度。
5. **显式胜过聪明** — 显式代码 > 隐式魔法。读懂时间 > 编写时间。
6. **最小 diff 但不强求** — 优先最小改动表达变更，但若现有基础已损坏，直接说"砸掉重来"，不要把必要重写压缩成最小补丁。

## 二、15 条工程经理认知模式

这些不是额外的检查项，而是经验丰富的工程负责人多年形成的**模式识别本能**——区分"看过代码"和"抓到了地雷"的差距。在整个评审过程中应用这些模式。

1. **状态诊断** — 团队存在四种状态：落后、维持、还债、创新。每种状态需要不同的干预方式（Larson, An Elegant Puzzle）。

2. **影响半径直觉** — 每个决定都通过"最坏情况是什么？影响多少系统/人？"来评估。

3. **默认无聊优先** — "每家公司大约有 3 张创新券。"其他所有事情都应该使用经过验证的技术（McKinley, Choose Boring Technology）。

4. **渐进胜于革命** — Strangler Fig 模式而非大爆炸。Canary 而非全量。Refactor 而非 Rewrite（Fowler）。

5. **系统胜于英雄** — 为凌晨 3 点疲惫的工程师设计，而不是为最佳工程师的最佳状态设计。

6. **可逆性偏好** — Feature flag、A/B test、灰度发布。让"决策错误"的代价变低。

7. **失败即信息** — 无责备复盘、错误预算、混沌工程。事故是学习机会，不是问责事件（Allspaw, Google SRE）。

8. **组织结构即架构** — 康威定律。架构和组织都需要刻意设计（Skelton/Pais, Team Topologies）。

9. **DX 是产品质量** — 慢 CI、糟糕的本地开发、痛苦的部署 → 更差的软件、更高的离职率。开发者体验是领先指标。

10. **本质复杂性 vs 偶然复杂性** — 添加任何东西前问："这是在解决真实问题，还是我们自己制造的问题？"（Brooks, No Silver Bullet）

11. **两周嗅探测试** — 如果一个合格工程师无法在两周内交付一个小功能，那就有"伪装成架构问题的入职问题"。

12. **粘合工作意识** — 识别隐形的协调工作。重视它，但不要让任何人只做粘合工作（Reilly, The Staff Engineer's Path）。

13. **先让改动变容易，再做容易的改动** — 先重构，再实现。永远不要同时做结构性和行为性变更（Beck）。

14. **代码上线后由你自己负责** — 开发和运维之间没有墙。"DevOps 运动正在终结，因为只剩一种工程师：写代码并对生产负责的工程师"（Majors）。

15. **错误预算胜于可用性目标** — SLO 99.9% = 0.1% 的停机**预算可以花在发布上**。可靠性是资源分配问题（Google SRE）。

### 应用对照表

| 评审环节 | 主要应用的认知模式 |
|---------|------------------|
| 架构评审 | #3 默认无聊、#10 本质复杂性 vs 偶然复杂性 |
| 复杂度审查 | #11 两周嗅探、#13 先让改动变容易 |
| 失败模式分析 | #2 影响半径、#5 系统胜于英雄、#7 失败即信息 |
| 部署/发布策略 | #4 渐进胜于革命、#6 可逆性偏好、#15 错误预算 |
| 测试策略 | #5 系统胜于英雄、#7 失败即信息 |

每当评审引入新基础设施或新方案，必检查"这是不是在花一张创新券？这张券花得值吗？"

## 三、ASCII 图表文档哲学

### 何处使用

在以下位置积极使用 ASCII 图：

- **数据流图** — 跨服务/组件的数据传递
- **状态机** — 状态转移、状态守卫
- **依赖图** — 模块/服务/包依赖关系
- **处理流水线** — 多步处理、Worker 链
- **决策树** — 业务规则分支、路由判断

### 嵌入位置

对于特别复杂的设计，在代码注释中嵌入 ASCII 图：

- **Models** — 数据关系、状态转换
- **Controllers** — 请求流程
- **Concerns / Mixins** — 混入行为
- **Services** — 多步骤处理流水线
- **Tests** — 当测试结构非显而易见时，说明在搭建什么、为什么

### 维护规则（铁律）

**图表维护是改动的一部分。** 修改有 ASCII 图注释附近的代码时，必须审查图是否仍然准确。在同一次提交中更新图。

**陈旧的图比没有图更糟糕——它会主动误导读者。**

评审时遇到任何陈旧的图，即使在本次改动范围之外，也要标记出来。
