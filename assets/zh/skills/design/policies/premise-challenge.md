# Premise Challenge — 前提挑战

## 核心原则

设计阶段必须主动识别并挑战用户描述中的**未验证前提**。前提 ≠ 事实，除非有证据支撑。

挑战的目的不是攻击用户，而是把"被当作事实的假设"显式化，让任一前提不成立时设计方案能及时调整。**语气必须合作式，禁止做成攻击**。

---

## 一、什么是"未验证前提"

用户陈述中被当作事实但实际未经证实的假设，例如：
- "用户不会做 X" — 如何确认？
- "这个 API 性能足够" — 有基准测试吗？
- "现有系统不支持 Y" — 验证过吗？
- "这是唯一的方案" — 真的搜索过替代方案？

---

## 二、前提选取规则

提炼前提时**必须**覆盖以下类型中的至少 3 类（不强求全覆盖）：

| 类型 | 例子 |
|------|------|
| **范围前提** | "本次仅改造下单主链路，购物车与售后链路不在范围内" |
| **依赖前提** | "假设上游订单中心的 `order_id` 字段在本季度内不会扩位或变更类型" |
| **数据/规模前提** | "假设每日新增审计日志 ≤ 500MB，超出该规模需重新评估存储方案" |
| **用户行为前提** | "假设运营人员会在导入失败后 24h 内查看告警邮件，而非依赖工单催办" |
| **技术选型前提** | "假设继续基于 Redis 6.x 做幂等键存储，不引入 KV 数据库或外部锁服务" |
| **优先级前提** | "数据一致性优先于写入吞吐，可接受 P99 延迟从 50ms 升到 150ms" |

每条前提**必须满足**：单句、可证伪、不成立时影响明确。**禁止**写成"用户会同意 X"这种空话。

每次提炼 **3-5 条**前提呈现给用户。

---

## 三、输出格式

```
🧩 Premise Challenge（前提挑战）

请逐条确认以下前提是否成立。任何一条不成立都会改变设计方案。

PREMISES:
1. <前提陈述，单句、可证伪> — agree / disagree / unsure
   类型：<范围 / 依赖 / 数据规模 / 用户行为 / 技术选型 / 优先级>
   风险：<高 / 低>（高 = 不成立会导致架构返工；低 = 影响有限）
   依据：<这条前提是从对话哪里推出来的>
   不成立的影响：<如果不成立，设计的哪个部分需要改>
   验证方式：<如何确认这条前提，例如基准测试 / 抽样调研 / 查阅文档>

2. <前提陈述> — agree / disagree / unsure
   ...

3. <前提陈述> — agree / disagree / unsure
   ...
```

---

## 四、用户回复处理

| 用户回复 | 后续动作 |
|---------|---------|
| 全部 agree（或"全部确认"） | 进入"完整方案确认询问"阶段 |
| 任意 disagree | 针对该条回到讨论：让用户给出修正后的前提；修正后**必须**重新生成 Premise Challenge 清单（**最多 3 轮**） |
| 任意 unsure | AI 必须给出具体追问（具体例子 / 边界场景 / 可验证标准），帮用户从 unsure → agree/disagree |
| 模糊回复（"差不多"/"大概吧"） | 视为 unsure，按上一行处理 |

---

## 五、阶段产出

通过的所有前提**必须**在 `pre_design.md` 中以 `## Premises` 节单独列出（位于 `## Decisions` 节之前），格式：

```markdown
## Premises
<!-- 进入实施前已与用户逐条确认的前提，任一不成立会改变设计 -->

1. <前提原文>
2. <前提原文>
3. <前提原文>
```

被用户拒绝 / 修改过的前提历史以单独的 `## Premise History` 节写入 `pre_design.md`，作为 lock 阶段评审建议落地时的"假设变更日志"参考。

---

## 六、唯一禁止行为

**禁止把挑战做成攻击**——语气必须合作式，对事不对人。
