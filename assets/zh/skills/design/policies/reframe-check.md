# Reframe Check — 重 framing 候选

## 核心原则

在为用户呈现具体设计方案之前，必须先检查"用户原始诉求"是否真的是要解决的问题。Reframe 是**问题层**的重新切割，不是实现层的多选。

> ⚠️ 本文件包含两节、顺序固定：
> - **第 1 节：Reframe 候选**（问题层重 framing）
> - **第 2 节：设计决策方案 Options**（实现层多方案对比）
>
> 必须先做第 1 节、再做第 2 节。两者解决的不是同一类问题：Reframe 问"是不是要解决另一个问题"，Options 问"已确定的问题用哪条路径实现"。

---

## 1. Reframe 候选输出格式（问题层）

### 1.1 何时可跳过 Reframe（唯一判定门）

仅当以下 **全部** 满足时，AI 可以跳过 Reframe 候选并明确说明 "本次需求 framing 已足够清晰，无需 Reframe"：

- 用户已自行说明了上下文 / 问题边界 / 受影响人群
- brainstorming 中所有具体追问都已得到具体答案（**无模糊回答**）
- 不存在比当前 framing 更小或更大的合理替代切片

否则**必须**输出至少 1 个 Reframe 候选并等待用户在 A/B/C 之间做出选择。

### 1.2 输出模板

```
🔍 Reframe 候选 — 这是不是另一个问题？

[原始问题]  <用 1 句话复述用户的原始诉求，必须用名词短语，不要带"做/实现"等动词形态>
     ↓ 我怀疑真正要解决的是 ↓
[更深问题]  <重新切割后的问题，可以更大 / 更小 / 换层级 / 换角色>

支撑这个 Reframe 的对话证据（1-3 处，必须引用用户原话）：
  · 「<用户原话片段 1>」
  · 「<用户原话片段 2>」
  · 「<用户原话片段 3>（如有）」

如果 Reframe 命中，方案的差异（一句话）：
  原方向 → <用户原方案的动词短语>
  新方向 → <Reframe 后会变成的动词短语>

请用以下任一选项回复：
  ✅  命中——按新问题继续设计
  ✏️  部分对，需要补充：<请直接写出补充内容>
  ❌  不需要 reframe，按原始诉求继续
```

### 1.3 用户选择处理

| 选择 | 后续动作 |
|------|---------|
| ✅ 命中 | 用 Reframe 后的理解作为后续设计起点 |
| ✏️ 部分对 | 把用户补充的内容合并进 Reframe，再发一次 Reframe 候选给用户确认（**最多 2 轮**） |
| ❌ 不需要 reframe | 保留原始 framing，记录"用户拒绝 Reframe" |

> 用户回复未使用上述符号但语义明确（如"对"、"就是这个意思"→ ✅；"差不多但 X 要改"→ ✏️；"不用，按原来的"→ ❌）按对应分支处理。语义不明确时必须重新呈现模板让用户在三选项中明确回复，禁止主观判定。

### 1.4 阶段产出

Reframe 阶段必须在 `pre_design.md` 中记录一行 **"Reframe 历程"**，例如：
- `原始诉求 X → 用户接受 Reframe 为 Y`，或
- `用户拒绝 Reframe，保留原始 framing X`

## 2. 设计决策方案输出格式（Options，实现层）

### 2.1 触发规则

每个**实现层决策点**必须提供至少 2-3 个备选方案，附带各自的优劣权衡。**禁止单方案直接推进**。

跳过条件（满足任一即可跳过 Options）：
- 用户明确说"不要给替代方案，就用这个"
- 该决策点已在前序对话中充分讨论过具体方案对比
- 纯机械操作无设计决策（如"创建目录"）

### 2.2 输出模板

```markdown
### 方案 A：{{NAME}}（推荐/不推荐）
- 优点：…
- 缺点：…
- 适用场景：…

### 方案 B：{{NAME}}
- 优点：…
- 缺点：…
- 适用场景：…

### 方案 C：{{NAME}}（如适用）
- 优点：…
- 缺点：…
- 适用场景：…

### 推荐选择：方案 X，因为 {{REASON}}
```

### 2.3 与 Premise Challenge 的衔接

被淘汰的方案及拒绝理由会在 Premise Challenge 完成后写入 `pre_design.md` 的 `## Alternatives` 节。
