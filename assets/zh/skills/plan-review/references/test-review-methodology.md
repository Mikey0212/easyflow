# 测试评审方法论

本文档是 plan-review skill 在 Section 3「测试评审」中的详细执行规范。

测试评审的目标是 **100% 覆盖**。评估计划中的每条代码路径，确保每条路径都有对应的测试。如果计划缺少测试，就把测试加到计划里——计划应足够完整，使得实施阶段从一开始就包含完整测试覆盖。

## Step 1: 检测测试框架

在分析覆盖率之前，先检测项目的测试框架：

1. **优先读取项目根目录的项目说明文件**（如 README、CONTRIBUTING、CLAUDE.md 等），寻找 `## Testing` 章节获取测试命令和框架名。如有，以此为权威来源。

2. **如未找到，自动检测**：
   - `Gemfile` → ruby（RSpec/Minitest）
   - `package.json` → node（Jest/Vitest/Mocha/Playwright）
   - `requirements.txt` 或 `pyproject.toml` → python（pytest/unittest）
   - `go.mod` → go（testing）
   - `Cargo.toml` → rust（cargo test）
   - `pom.xml` 或 `build.gradle` → java（JUnit/TestNG）

3. **检查现有测试基础设施**：
   - 配置文件：`jest.config.*`、`vitest.config.*`、`playwright.config.*`、`cypress.config.*`、`.rspec`、`pytest.ini`、`phpunit.xml`
   - 测试目录：`test/`、`tests/`、`spec/`、`__tests__/`、`cypress/`、`e2e/`

4. **如果完全没有测试框架**：仍然产出覆盖率图，但跳过具体测试代码生成，只在评审报告中记录"项目无测试框架，建议先引入"。

## Step 2: 追踪计划中每条代码路径

读取计划文档。对每个新功能、服务、端点或组件，追踪数据将如何流经代码——不要只列出计划中的函数，要**实际跟着计划好的执行路径走一遍**。

### 2.1 读取计划

对每个计划组件，理解它做什么、如何与现有代码连接。

### 2.2 追踪数据流

从每个入口点（路由处理器、导出函数、事件监听器、组件渲染）开始，跟随数据穿过每一个分支：

- **输入来自哪里**：request params、props、数据库、API 调用？
- **谁在转换它**：校验、映射、计算？
- **数据去哪里**：数据库写入、API 响应、UI 渲染、副作用？
- **每一步可能出什么错**：null/undefined、非法输入、网络故障、空集合？

### 2.3 绘制执行图

对每个改动文件，画 ASCII 图，展示：

- 每个新增或修改的函数/方法
- 每个条件分支（if/else、switch、三元、guard、early return）
- 每条错误路径（try/catch、rescue、error boundary、fallback）
- 每次调用其他函数（继续追踪进去——它有未测试的分支吗？）
- 每条边：null 输入怎样？空数组？非法类型？

**这是关键步骤**——你正在构建一张图，显示每行可以根据输入而不同执行的代码。**这张图里的每个分支都需要一个测试。**

## Step 3: 映射用户流、交互、错误状态

代码覆盖不够——需要覆盖**真实用户如何与变更后的代码交互**。对每个变更功能，思考：

### 3.1 用户流（User Flows）

用户的什么操作序列会触及这段代码？完整地画出旅程（例如"用户点击 Pay → 表单校验 → API 调用 → 成功/失败页面"）。**旅程的每一步都需要一个测试。**

### 3.2 交互边界情况

用户做了意料之外的事会怎样？

- **快速重复操作**：双击、快速连击、表单连续提交
- **导航中断**：中途按返回键、关标签、点别的链接
- **陈旧数据提交**：页面开了 30 分钟、session 过期
- **慢连接**：API 花 10 秒——用户看到什么？
- **并发操作**：两个标签页、同一份表单

### 3.3 用户可见的错误状态

代码处理的每个错误，用户实际经历什么？

- 是清晰的错误提示，还是静默失败？
- 用户能恢复（重试、回退、修正输入）还是被卡住？
- 没网络时呢？API 返回 500 时呢？服务器返回非法数据时呢？

### 3.4 空/零/边界状态

- 零结果时 UI 显示什么？
- 10000 条结果时呢？
- 单字符输入时？
- 最大长度输入时？

把这些加到执行图旁边。**没有测试的用户流和没有测试的 if/else 是同等大小的缺口。**

## Step 4: 对照现有测试逐分支检查

逐分支过你的图——既包括代码路径**也包括用户流**。对每条分支，搜索测试：

- 函数 `processPayment()` → 找 `billing.test.ts`、`billing.spec.ts`、`test/billing_test.rb`
- if/else → 找 true 和 false 两条路径都有测试
- 错误处理器 → 找触发那个特定错误条件的测试
- 调用 `helperFn()` 且它有自己的分支 → 那些分支也需要测试
- 用户流 → 找走完整个旅程的集成或 E2E 测试
- 交互边界情况 → 找模拟意外操作的测试

### 测试质量评分

| 标记 | 含义 |
|-----|------|
| ★★★ | 测试了行为 + 边界情况 + 错误路径 |
| ★★ | 测试了正确行为，只走 happy path |
| ★ | 烟雾测试 / 存在性检查 / 无意义断言（如"它能渲染"、"它不抛错"） |

## Step 5: E2E vs Unit 决策矩阵

对每条分支，判断该用单元测试还是 E2E/集成测试。

### 推荐 E2E（在图中标 [→E2E]）

- 涉及 3+ 组件/服务的常见用户流（例如：注册 → 邮箱验证 → 首次登录）
- 集成点，mock 会掩盖真实失败（例如：API → 队列 → worker → DB）
- 认证/支付/数据销毁流程——太重要，不能只信单元测试

### 推荐 EVAL（在图中标 [→EVAL]）

- 关键 LLM 调用，需要质量评估（例如：prompt 改动 → 测试输出仍达质量基线）
- prompt 模板、系统指令、tool 定义的变更

### 保留单元测试

- 输入输出清晰的纯函数
- 无副作用的内部 helper
- 单个函数的边界情况（null 输入、空数组）
- 不面向用户的冷僻流程

## 回归测试铁律（IRON RULE）

**当覆盖率审计识别出"回归"——之前能工作但本次 diff 弄坏的代码——必须把回归测试作为关键需求加入计划。不发起 ask_followup_question。不跳过。**

回归测试是最高优先级的测试，因为它证明某个东西确实坏了。

### 什么是回归

- diff 修改了**已有行为**（不是新代码）
- 现有测试（如有）未覆盖被改动的路径
- 改动给现有调用者引入了新的失败模式

**判断不确定时，默认写回归测试。** 宁可多写。

在同一张图里包含代码路径和用户流。标记 E2E 候选和 eval 候选：

### 快速通道

如果所有路径都已覆盖：输出"测试评审：所有新代码路径均有测试覆盖 ✓"，继续下一节。

## Step 7: 把缺失的测试加入计划

对图中识别出的每个 GAP，把测试需求加入计划。要具体：

- **测试文件路径**：匹配现有命名约定
- **测试要断言什么**：具体输入 → 预期输出/行为
- **测试类型**：单元 / E2E / eval（用决策矩阵）
- **回归测试**：标记为 **CRITICAL** 并解释什么坏了

计划应足够完整，使得实施阶段每个测试都与功能代码同步写——而不是推迟到后续 PR。

## 产物写入约定（单一产物原则）

测试评审的产物**直接写入** review-report.md 的「测试评审」章节，**不另产出 test-plan.md 文件**。

- **OpenSpec 场景**：写到 `openspec/changes/<name>/review-report.md` 的「测试评审」节，符合 OpenSpec 单一产物原则
- **独立调用场景**：写到调用方指定的 review-report.md，路径由调用方决定

结构如下：

```markdown
## 3. 测试评审

### 测试框架
- 检测到: [framework name + version]
- 命令: `[test command]`

### 覆盖率图
[ASCII 图，见上文 Step 6 格式]

### 缺口列表（按优先级）

#### CRITICAL（回归测试，必须加入 tasks.md）
- [F1] [文件:行号] — [描述]

#### P1（P1 级别，影响合并）
- ...

#### P2（可推迟，加入 TODO）
- ...

### 加入 tasks.md 的测试任务

- [ ] T-test-001: 在 `<file>` 中添加 `<test name>` 测试 [回归]
- [ ] T-test-002: 在 `<file>` 中添加 `<test name>` 测试 [E2E]
- ...
```

**调用方约束**：测试缺口转换为 tasks.md 中的可执行任务这一步**由调用方在下一阶段启动前统一处理**（在 easy-flow lock 链中即由 `/ezfl:build` 入口处理），本 skill 在 review-report.md 中列出建议即可，**禁止直接修改 tasks.md**。
