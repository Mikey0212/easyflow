# 输出风格守则与必需输出格式

本文档定义 plan-review skill 的输出风格、置信度规则、ask_followup_question 格式约束，以及评审结束时必需的产物。

## 一、输出风格守则

### 1. 问题用结果导向描述，不用实现导向

每次发起 ask_followup_question 时，优先描述**用户/系统会经历什么**，而不是**底层实现是什么**。

**反例**（实现导向，用户疲劳）：
> "这个端点是否幂等？"
> "数据库使用悲观锁还是乐观锁？"
> "是否需要分布式事务？"

**正例**（结果导向，用户秒懂）：
> "如果用户双击按钮，操作执行两次可以吗？"
> "如果两个人同时编辑同一条记录，谁的改动会被保留？"
> "如果中间一步失败了，前面已经做的事会自动撤销吗？"

### 2. 每个决策都连接到具体用户影响

每个 ask_followup_question 选项必须有一句"如果选这个，后果是什么"。

**反例**：
> "这可能会比较慢"

**正例**：
> "这是 N+1 查询。50 条数据时，每次列表加载会增加 200ms，用户会看到列表卡顿一下"

### 3. 短句、具体名词、主动语态

- ❌ "应当对边缘情形进行充分的考虑与处理"
- ✅ "处理这 3 个边界：null 输入、空数组、超长字符串"

### 4. 概要中文化

在中文环境下不需要英文术语 gloss 表。如果输出中出现以下术语，**第一次出现时**附一句中文括注：

| 术语 | 中文括注示例 |
|------|------------|
| 幂等 / idempotent | （重复调用结果一致） |
| race condition | （两件事同时发生互相踩脚） |
| N+1 查询 | （一次本可以拉完的数据，被拆成 N 次查询） |
| 雪崩 / cache stampede | （缓存同时失效，流量直冲数据库） |
| backpressure | （下游处理不过来时，反推上游减速） |

不在此列表的术语默认假定读者懂，无需解释。

## 二、置信度校准

每条 finding 必须带置信度分数（1-10）：

| 分数 | 含义 | 显示规则 |
|-----|------|---------|
| 9-10 | 已读具体代码验证。具体 bug 或漏洞可演示。 | 正常显示 |
| 7-8 | 高置信度模式匹配。极可能正确。 | 正常显示 |
| 5-6 | 中等置信。可能是误报。 | 显示并加备注："中置信，需验证是否真问题" |
| 3-4 | 低置信。模式可疑但可能没事。 | **不进主报告**，只进附录 |
| 1-2 | 推测。 | 仅当严重度为 P0 时才报告 |

### Finding 格式

```
[严重度] (置信度: N/10) file:line — 描述
```

**示例**：

```
[P1] (置信度: 9/10) src/auth/login.ts:42 — SQL 注入，where 子句使用字符串拼接
[P2] (置信度: 5/10) src/api/users.ts:18 — 疑似 N+1 查询，需用生产日志验证
```

### 校准学习

如果 confidence < 7 的 finding 被用户确认是真问题，这是一次**校准事件**——你的初始置信度太低。把修正后的模式记入项目记忆（memory/），供未来评审用更高置信度抓到。

## 三、ask_followup_question 格式约束

### 一问一议（One issue = One question）

每个发现单独发起一次 ask_followup_question，**绝不打包**。

理由：用户面对一次 5 个问题会疲劳决策，倾向不看细节直接 yes/no。每个问题展开 2-3 个具体选项，用户能在 5 秒内选完。

### 标准结构

每次 ask_followup_question 必须包含 4 部分：

1. **重新定位**（1-2 句）：说明项目、当前分支、当前任务。让用户即使 20 分钟没看这个窗口也能秒懂。

2. **简化问题**（plain English，约 16 岁的人能懂）：用具体例子和类比。说"它做什么"，不说"它叫什么"。如果你需要读源码才能解释自己的话，说明太复杂了。

3. **推荐**（必带格式）：
   ```
   推荐: 选 [字母] 因为 [一句话理由，映射到工程偏好的某一条]
   完整度: A=X/10, B=Y/10
   ```

4. **选项**（2-3 个，字母编号）：
   ```
   A) [描述] (人工: ~X / AI辅助: ~Y) Completeness: N/10
   B) [描述] (人工: ~X / AI辅助: ~Y) Completeness: M/10
   ```

### 完整度评分（Completeness Principle）

每个选项标 `Completeness: X/10`：

- **10** = 完整实现（所有边界 + 全覆盖 + 错误路径全处理）
- **7** = 仅覆盖 happy path
- **3** = 推迟大量工作的捷径

**总是推荐 ≥8 的选项**。如果两个都 ≥8，推荐高的。如果有选项 ≤5，显式标记为"shortcut"并说明被推迟的工作量。

### 效率参考表（Completeness Principle 的另一面）

| 任务类型 | 人类团队 | AI 辅助 | 压缩比 |
|---------|---------|---------|--------|
| 样板代码 | 2 天 | 15 分钟 | ~100x |
| 测试 | 1 天 | 15 分钟 | ~50x |
| 功能 | 1 周 | 30 分钟 | ~30x |
| Bug 修复 | 4 小时 | 15 分钟 | ~20x |

AI 辅助让"完整性"几乎免费。永远推荐**完整方案**而不是捷径，因为差距常常只有几分钟。**如果一个完整方案只比捷径多花几分钟（用 AI 辅助），必须推荐完整方案。**

## 四、必需输出三件套

评审结束后，必须产出以下三个章节，缺一不可。这些章节都写入 `review-report.md`（OpenSpec 场景下为 `openspec/changes/<name>/review-report.md`）。

### 1. NOT in scope（明确不在范围内）

列出**评审期间考虑过但被显式推迟**的工作，每条带一句 rationale。

格式：

```markdown
## NOT in scope

本次提案考虑过但不包含以下工作：

- **[工作项 1]** — [一句话理由，例如"需独立的需求挖掘，推到下一迭代"]
- **[工作项 2]** — [理由]
```

**为什么必需**：防止"模糊推迟"。每个被推迟的事都必须被命名，否则会以"忘了"的形式出现在生产事故里。

### 2. What already exists（已有代码盘点）

列出**已经部分或完全解决子问题的现有代码/流程**，以及计划是复用了它们，还是不必要地重建。

格式：

```markdown
## What already exists

| 子问题 | 现有方案 | 计划处理方式 | 评价 |
|--------|---------|-------------|------|
| 用户认证 | `auth/jwt-validator.ts` 已实现 JWT 校验 | 复用 | ✅ |
| 速率限制 | `middleware/rate-limit.ts` 有基础设施 | 重建一份 | ❌ 应复用，标 P1 |
```

**为什么必需**：防止 AI 辅助时代的"信息过载偏向重建"。AI 写新代码很容易，导致默认重建而不是复用。

### 3. Failure modes（失败模式）

对测试评审图中识别出的每条新代码路径，列出一个真实的生产失败方式（timeout / nil / race / 数据陈旧），并标注：

1. 是否有测试覆盖该失败
2. 是否有错误处理
3. 用户会看到清晰错误还是静默失败

格式：

```markdown
## Failure modes

| 代码路径 | 真实失败方式 | 有测试？ | 有错误处理？ | 用户可见？ |
|---------|-------------|--------|-----------|----------|
| `processPayment()` | 第三方 API 超时 5 秒 | ❌ | ⚠️ 仅 catch 不重试 | ❌ 静默失败 |
| `refundPayment()` | 部分退款金额非法 | ✅ | ✅ | ✅ |

### 关键缺口（critical gaps）

[列出 "无测试 + 无错误处理 + 静默失败" 三个条件全中的项，标记为 critical gap]
- `processPayment()` API 超时 — **CRITICAL GAP**: 用户付完钱看不到任何反馈，以为支付未发生会重试
```

**为什么必需**：任何"无测试 + 无错误处理 + 静默失败"的失败模式都是 **critical gap**，必须在实施前修复。

## 五、Worktree 并行化策略（可选输出）

仅在评审检测到**多条独立工作流**时产出。

### 跳过条件

如果**所有步骤都触及同一主模块**，或**计划只有少于 2 条独立工作流**，跳过此节，写一句："顺序实施，无并行机会。"

### 输出格式

#### 5.1 依赖表

每条实施步骤/工作流：

| 步骤 | 触及模块 | 依赖于 |
|------|---------|--------|
| T1: 添加 API 端点 | `controllers/` `models/` | — |
| T2: 添加前端 | `web/components/` | T1 完成接口约定 |
| T3: 添加 worker | `workers/` `models/` | T1（共享 model） |

工作在**模块/目录级别**，不在文件级别。文件级别是猜测，模块级别是可靠的。

#### 5.2 并行 Lane 分组

- 无共享模块且无依赖 → 不同 Lane（可并行）
- 共享模块目录 → 同一 Lane（顺序执行）
- 有依赖关系 → 排在后面的 Lane

格式：

```
Lane A: T1 → T3（顺序，共享 models/）
Lane B: T2（独立，等 T1 完成接口后启动）
```

#### 5.3 执行顺序

```
1. 先启动 Lane A 第一步（T1）
2. T1 完成接口契约后，Lane B（T2）与 Lane A 第二步（T3）并行
3. 全部 merge
```

#### 5.4 冲突标记

如果两条并行 Lane 触及同一模块目录，标记：

> ⚠️ Lane A 与 Lane C 都触及 `models/` —— 潜在 merge 冲突。考虑顺序执行或精细协调。

## 六、Completion Summary（评审完成摘要）

评审结束在报告末尾填写并显示：

```markdown
## 评审完成摘要

| 项目 | 状态 |
|------|------|
| Step 0 范围挑战 | [范围按原提案接受 / 范围已按建议缩减] |
| 架构评审 | ___ 个问题已发现 |
| 代码质量评审 | ___ 个问题已发现 |
| 测试评审 | 覆盖率图已产出，___ 个 GAP 已识别 |
| 性能评审 | ___ 个问题已发现 |
| NOT in scope | 已写入 |
| What already exists | 已写入 |
| Failure modes | ___ 个 critical gap 已标记 |
| Outside Voice | [已运行: cross-review-agent（模型: <name>） / 已跳过 / 已降级为 inline] |
| Worktree 并行化 | ___ Lane，___ 并行 / ___ 顺序（或: 顺序实施） |
| Completeness Score | X/Y 推荐项选了完整方案 |

**未解决的决策（可能未来反咬你一口）**：
- [若用户未答某 ask_followup_question 或打断，在此列出]

## STATUS

[四选一，见 Escalation 升级协议]
- DONE — 全部完成，每个声明都有证据
- DONE_WITH_CONCERNS — 完成但有问题需用户知晓 [列出每个 concern]
- BLOCKED — 无法继续 [说明卡在哪，试过什么]
- NEEDS_CONTEXT — 缺信息无法继续 [说明需要什么]
```

## 七、未解决决策处理（Unresolved decisions）

如果用户：

- 没回答某个 ask_followup_question
- 打断让你继续
- 跳过某节评审

→ **绝不静默默认某选项**。在最后的 Completion Summary 中单列「未解决的决策」章节，列出每条未解决项及对应可能后果。

格式：

```markdown
**未解决的决策**：
1. [问题简述] — 用户跳过未答。可能后果：[一句话]
2. [问题简述] — 用户打断。可能后果：[一句话]
```

## 八、Escalation 升级协议（Completion Status Protocol）

**核心理念**：烂工作比没工作糟。你不会因为升级被惩罚。

### 何时必须升级

- 同一任务尝试 3 次失败 → STOP
- 安全敏感修改（认证、加密、权限、PII）不确定 → STOP
- 工作范围超出可验证能力 → STOP

### 升级格式

```markdown
## STATUS: BLOCKED / NEEDS_CONTEXT

**REASON**: [1-2 句]
**ATTEMPTED**: [试过什么]
**RECOMMENDATION**: [用户该做什么]
```

把 STATUS 直接写入 review-report.md 末尾。**不要为了"显得专业"而硬撑出低质量评审报告**——明确说"这超出我能力范围"才是专业。
